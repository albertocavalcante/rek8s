# GKE Gateway Blueprint

This Terraform root creates a GKE cluster shaped for the
[`gke.yaml`](../../cluster-profiles/gke.yaml) `rek8s` profile.

It provisions:

- a dedicated VPC and subnetwork
- secondary IP ranges for pods and services
- a regional GKE Standard cluster
- Dataplane V2
- Gateway API support
- GCE PD CSI support
- a worker node pool sized for a first `rek8s` deployment

It does not install `rek8s`, `cert-manager`, or the `Gateway` object. Those
remain separate so the Terraform root stays cluster-focused and easy to reason
about.

## Usage

```bash
cp terraform.tfvars.example terraform.tfvars
terraform init
terraform plan
terraform apply
gcloud container clusters get-credentials <cluster-name> --region <region> --project <project-id>
kubectl apply -f ../../gateways/gke-external-gateway.yaml
helm install rek8s ../../../charts/rek8s \
  -f ../../cluster-profiles/gke.yaml \
  --set global.domain=build.example.com \
  --set cluster.ingress.gatewayRef.name=rek8s-public \
  --set cluster.ingress.gatewayRef.namespace=rek8s-ingress
```

## Main GKE Gotchas

- Dataplane V2 already provides the standard `NetworkPolicy` behavior that the
  `rek8s` GKE profile expects. Do not re-enable the legacy GKE network-policy
  add-on on top of it.
- Gateway API support on the cluster does not create the `Gateway` resource for
  you. `rek8s` still expects that object to exist.
- The Gateway listener certificate and the per-service TLS secrets generated by
  `cert-manager` are separate layers.
